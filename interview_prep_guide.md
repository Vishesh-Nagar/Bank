# 🚀 The Bank Microservices Project: Ultimate Interview Guide

This document is your definitive guide to understanding every architectural decision, design pattern, and trade-off in the Bank project. It is designed to prepare you for the most rigorous System Design and Backend Engineering interviews at top-tier tech companies.

---

## 1. High-Level Architecture Overview

The system is built as a **Distributed Microservice Architecture** using Java 21 and Spring Boot 3. 

### The Services
1. **API Gateway (`api-gateway`)**: The single entry point. Handles dynamic routing, CORS, JWT termination, and custom Distributed Rate Limiting.
2. **User Service (`user-service`)**: Manages identity. Handles registration, email verification (via SMTP), password resets, and issues JWT Access/Refresh tokens.
3. **Account Service (`account-service`)**: The core financial ledger. Manages account balances, deposits, withdrawals, and implements the **Transactional Outbox Pattern** for guaranteed event delivery.
4. **Payment Service (`payment-service`)**: Orchestrates money movement between accounts. Enforces idempotency (to prevent double-charges) and daily transfer limits using atomic Redis operations.
5. **Notification Service (`notification-service`)**: A WebSocket server that consumes Kafka events and pushes real-time alerts to the frontend client.

### The Infrastructure
- **MySQL**: The primary relational datastore. Each microservice has its own logically isolated database (`bank_users`, `bank_accounts`, `bank_payments`) to enforce loose coupling.
- **Redis**: Used for high-speed, transient data: JWT blocklists, Idempotency keys, Transfer Limits, and the Sliding Window Rate Limiter.
- **Kafka**: The central nervous system. Facilitates asynchronous event-driven communication (e.g., the Account Service broadcasting a balance change so the Notification Service can alert the user).
- **Zipkin**: The observability platform. Collects OpenTelemetry spans to visualize network latency across the cluster.

## 2. Senior-Level Architectural Patterns (The Resume Boosters)

These are the features that elevate this project from a standard CRUD app to a Big Tech, production-ready system. 

### A. The Transactional Outbox Pattern (Solving the Dual-Write Problem)
**The Problem:** In a distributed system, a service often needs to update the database AND publish an event to Kafka. If you update the database and then call `kafkaTemplate.send()`, the network to Kafka might drop. Your database commits, but the event is lost. Downstream services never find out about the change. This is the "Dual-Write Problem".
**Your Solution:** 
- In the `account-service`, when a balance changes, you save the new balance AND an `OutboxEvent` record to an `outbox_events` table within the *exact same ACID database transaction*. 
- If either fails, both roll back. If it succeeds, the event is guaranteed to be persisted.
- A background daemon (`OutboxRelayJob`) continuously polls the `outbox_events` table and reliably publishes the payloads to Kafka, marking them `COMPLETED` only upon Kafka's acknowledgement.
- **Interview Buzzwords:** Atomicity, At-Least-Once Delivery Guarantees, Decoupling Database I/O from Network I/O.

### B. Custom Distributed Sliding Window Rate Limiter
**The Problem:** Standard "Fixed Window" rate limiters (e.g., 100 requests per minute) suffer from "boundary bursts." A user can send 100 requests at 12:59:59 and another 100 at 1:00:01, effectively sending 200 requests in 2 seconds.
**Your Solution:** 
- You ripped out the default Spring Cloud Gateway limiter and implemented the **Sliding Window Log** algorithm from scratch.
- You wrote a custom Lua script that executes atomically inside Redis using Sorted Sets (`ZSET`).
- The script automatically removes timestamps older than the window, counts the remaining requests (`ZCARD`), and either allows or denies the request, while setting a `PEXPIRE` (TTL) to prevent memory leaks for inactive IPs.
- **Interview Buzzwords:** Redis Sorted Sets, Lua Scripting for Atomicity, Boundary Burst Mitigation, O(N) memory complexity optimization.

### C. Idempotency (Safe Retries)
**The Problem:** If a user submits a $500 payment and their mobile network drops while waiting for the HTTP 200 OK, they might click "Submit" again, resulting in a duplicate $500 charge.
**Your Solution:**
- The `payment-service` requires an `Idempotency-Key` header (usually a UUID generated by the frontend).
- You use Redis to check if this key has been seen in the last 24 hours.
- If it's a new key, you process the payment and store the resulting `paymentId` in Redis. 
- If the user retries with the same key, you short-circuit the logic and return the original cached response, guaranteeing the payment is only processed exactly once.
- **Interview Buzzwords:** Exactly-Once Processing Semantics, Network Partition Tolerance, Safe Retries.

### D. Concurrency Control (Race Conditions)
**The Problem:** Two threads attempt to deduct $100 from an account with a $150 balance at the exact same millisecond. Both read $150, both do math, both save $50. You lost a transaction.
**Your Solution:**
- **Database (Pessimistic Locking):** In `AccountServiceImpl`, you use `findByIdForUpdate()`. This executes `SELECT ... FOR UPDATE` in MySQL, locking the row exclusively until the transaction commits, forcing concurrent threads to queue up.
- **Redis (Atomic Lua):** For the Daily Transfer Limit, you cannot use a `GET` followed by a `SET` in Java, because another instance might update the limit in between. You wrote `transfer_limit.lua`, ensuring the read-and-increment happens as a single atomic operation in the single-threaded Redis engine.
- **Interview Buzzwords:** Pessimistic vs Optimistic Locking, Row-Level Locks, Redis Single-Threaded Event Loop.

### E. Observability & Distributed Tracing
**The Problem:** A user complains their payment failed, but all you have are gigabytes of unorganized text logs across 5 different services.
**Your Solution:**
- You integrated **Micrometer** and the **OpenTelemetry (OTel)** bridge into the Maven Parent POM.
- The API Gateway generates a globally unique `traceId` for every incoming HTTP request and propagates it downstream via B3 HTTP Headers.
- This `traceId` is injected into every SLF4J log statement via Mapped Diagnostic Context (MDC).
- Spans (timing data) are asynchronously exported to **Zipkin**, allowing you to visually identify network bottlenecks.
- **Interview Buzzwords:** Log Correlation, MDC, B3 Header Propagation, Telemetry, Mean Time to Recovery (MTTR).

### F. Circuit Breaker Pattern (Graceful Degradation)
**The Problem:** In a distributed system, if the `account-service` goes down, the API Gateway and `payment-service` will hang while waiting for HTTP timeouts. This consumes thread pools across the entire cluster, leading to a cascading failure that crashes healthy services.
**Your Solution:**
- You integrated **Resilience4j** to implement the Circuit Breaker pattern.
- The API Gateway wraps all outgoing routes in a circuit breaker. If a downstream service exceeds a 50% failure rate over 10 requests, the circuit "opens" and instantly routes to a `FallbackController`, returning a clean `503 Service Unavailable` JSON payload instead of a hanging stack trace.
- The `payment-service` also wraps its OpenFeign client to the `account-service` in a circuit breaker, allowing it to fail fast and preserve threads.
- **Interview Buzzwords:** Cascading Failures, Fail Fast, Graceful Degradation, Self-Healing, Half-Open State.

### G. 12-Factor App (Environment Configuration)
**The Problem:** Hardcoding database credentials or infrastructure URLs (e.g., `http://bank-user-service:8081`) into source code violates the 12-Factor App methodology. It prevents you from deploying the exact same compiled artifact across different environments (Dev, Staging, Prod).
**Your Solution:**
- You aggressively externalized all configuration. The Spring Boot `application.yaml` files now use environment variable placeholders with sensible defaults (e.g., `${USER_SERVICE_URL:http://bank-user-service:8081}`).
- The frontend Nginx container uses a custom `Dockerfile` that executes `envsubst` on startup, dynamically injecting the API Gateway URL into the Nginx configuration without hardcoding it.
- **Interview Buzzwords:** 12-Factor App, Infrastructure as Code, Stateless Configuration, Environment Parity.

### H. Kubernetes (Cloud-Native Orchestration)
**The Problem:** Docker Compose is great for local development, but in a production environment, you need a way to reliably scale individual microservices, handle self-healing (restarting crashed containers), and perform zero-downtime rolling updates.
**Your Solution:**
- You wrote a complete suite of Kubernetes manifests (`Deployments`, `Services`, `ConfigMaps`, `PersistentVolumeClaims`) to deploy the entire architecture to an orchestration cluster like AWS EKS or GKE.
- You used `ClusterIP` Services for internal Service Discovery, allowing the API Gateway to route to `http://bank-user-service:8081`, which Kubernetes' internal DNS load balances natively across multiple replicas of the user service.
- You used `ConfigMaps` and `Secrets` to securely inject the 12-Factor environment variables directly into the pods.
- **Interview Buzzwords:** Container Orchestration, Persistent Volumes, ClusterIP Load Balancing, Zero-Downtime Deployments, Pod Scheduling.

## 3. Potential Interview Questions & Answers

### Q: "Why did you choose a Microservices architecture instead of a Monolith?"
**A:** "For a banking application, bounded contexts are highly distinct. The User Service handles identity (high read throughput), while the Payment Service handles ledger math (high write consistency). Microservices allowed me to isolate the databases so a failure in the Notification Service wouldn't impact core payment processing. It also let me scale the Gateway independently to handle rate-limiting load."

### Q: "What happens if your Outbox Relay Job crashes halfway through processing a batch?"
**A:** "Because I use the Transactional Outbox pattern, the state is entirely durable in MySQL. If the job crashes, the events remain in the `PENDING` state. When the job restarts, it will simply pick them back up. Since I only mark an event `COMPLETED` *after* the Kafka `CompletableFuture.get()` returns successfully, a crash guarantees we never lose an event. In the worst case (it sent to Kafka but crashed before updating the DB), it will send a duplicate event on restart—which is fine, because my downstream consumers are designed to be idempotent."

### Q: "Why use a Lua script in Redis instead of just standard Java commands?"
**A:** "Redis is single-threaded. When I send a Lua script to Redis via `EVALSHA`, Redis executes the entire script atomically. No other command can interrupt it. If I did a `ZREMRANGEBYSCORE` and then a `ZCARD` from Java, a second instance of the API Gateway could interleave commands in between those two network round trips, leading to race conditions in the rate limiter."

### Q: "How does your system authenticate requests between microservices?"
**A:** "The API Gateway validates the JWT using the shared secret and strips the JWT from the request. It then injects a trusted `X-Authenticated-User` HTTP header containing the user's ID before proxying to downstream services. The downstream services trust this header. To prevent malicious users from bypassing the Gateway and hitting the microservices directly, I configured an internal shared secret (`INTERNAL_SERVICE_SECRET`) that is required for direct service-to-service communication (like the Account Service fetching user data from the User Service)."

### Q: "Why use Testcontainers for integration tests?"
**A:** "Using H2 in-memory databases for testing is an anti-pattern for complex systems because H2 doesn't have the same locking semantics, dialect, or features as MySQL (e.g., `FOR UPDATE` behaves differently). Testcontainers spins up an actual MySQL Docker image and a Redis image during the Maven `test` phase. This guarantees that my concurrency tests (`ConcurrentPaymentTest`) are running against production-equivalent infrastructure, eliminating 'it works on my machine' bugs."
